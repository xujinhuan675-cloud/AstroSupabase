# 搜索功能排查指南

## 问题描述
线上搜索接口 `/api/search` 始终返回空数组 `[]`，本地搜索正常。

## 可能的原因

### 1. 搜索向量未生成
**症状**: `search_vector` 字段为 NULL

**原因**:
- 迁移脚本未在线上数据库执行
- 触发器未创建或未生效
- 文章是在触发器创建之前插入的

**排查方法**:
```bash
npm run diagnose:search
```

**修复方法**:
```bash
npm run fix:search
```

### 2. 数据库配置问题
**症状**: 连接失败或权限不足

**排查**:
- 检查 `.env` 文件中的 `DATABASE_URL` 是否正确
- 检查 Vercel 环境变量是否配置
- 检查数据库用户权限

### 3. 全文搜索配置问题
**症状**: 查询语法错误或配置不匹配

**当前配置**:
- 使用 `simple` 配置（不分词）
- 使用 `plainto_tsquery` 函数
- 标题权重 'A'，内容权重 'B'

## 排查步骤

### 步骤 1: 运行诊断脚本
```bash
cd AstroSupabase
npm run diagnose:search
```

诊断脚本会检查：
1. 文章总数和已发布文章数
2. `search_vector` 字段是否存在且有值
3. 测试多个搜索查询
4. 检查触发器是否存在
5. 查看示例文章的搜索向量

### 步骤 2: 分析诊断结果

#### 情况 A: search_vector 全部为 NULL
**原因**: 触发器未创建或文章是旧数据

**解决方案**:
```bash
npm run fix:search
```

#### 情况 B: search_vector 有值但搜索无结果
**原因**: 可能是查询语法或配置问题

**检查**:
1. 查看诊断脚本中的测试查询结果
2. 检查搜索词是否与文章内容匹配
3. 尝试不同的搜索词

#### 情况 C: 本地正常但线上异常
**原因**: 环境差异

**检查**:
1. 线上数据库是否执行了迁移
2. 线上环境变量是否正确
3. 线上数据库版本是否支持全文搜索

### 步骤 3: 修复搜索索引

如果诊断发现 `search_vector` 为空，运行修复脚本：

```bash
npm run fix:search
```

这个脚本会：
1. 为所有文章重新生成 `search_vector`
2. 使用与触发器相同的逻辑
3. 更新所有现有文章

### 步骤 4: 验证修复

修复后，再次运行诊断：
```bash
npm run diagnose:search
```

或者直接测试 API：
```bash
# 本地测试
curl "http://localhost:4321/api/search?q=数学"

# 线上测试
curl "https://your-domain.vercel.app/api/search?q=数学"
```

## 手动修复步骤

如果自动脚本无法解决问题，可以手动执行以下 SQL：

### 1. 检查触发器
```sql
SELECT trigger_name, event_manipulation, action_statement
FROM information_schema.triggers
WHERE event_object_table = 'articles'
AND trigger_name = 'articles_search_update';
```

### 2. 创建触发器（如果不存在）
```sql
CREATE OR REPLACE FUNCTION articles_search_trigger() RETURNS trigger AS $
begin
  new.search_vector :=
    setweight(to_tsvector('simple', coalesce(new.title,'')), 'A') ||
    setweight(to_tsvector('simple', coalesce(new.content,'')), 'B');
  return new;
end
$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS articles_search_update ON articles;
CREATE TRIGGER articles_search_update BEFORE INSERT OR UPDATE
  ON articles FOR EACH ROW EXECUTE FUNCTION articles_search_trigger();
```

### 3. 重建所有文章的搜索向量
```sql
UPDATE articles 
SET search_vector = 
  setweight(to_tsvector('simple', coalesce(title,'')), 'A') || 
  setweight(to_tsvector('simple', coalesce(content,'')), 'B');
```

### 4. 验证结果
```sql
-- 检查有多少文章有搜索向量
SELECT 
  COUNT(*) FILTER (WHERE search_vector IS NOT NULL) as has_vector,
  COUNT(*) FILTER (WHERE search_vector IS NULL) as no_vector
FROM articles
WHERE is_deleted = false;

-- 测试搜索
SELECT id, title
FROM articles
WHERE is_deleted = false
  AND search_vector @@ plainto_tsquery('simple', '数学')
LIMIT 5;
```

## 常见问题

### Q1: 为什么使用 'simple' 配置而不是中文分词？
**A**: PostgreSQL 的 `simple` 配置不进行分词，适合：
- 中英文混合内容
- 避免复杂的分词配置
- 简化部署和维护

如果需要更好的中文搜索，可以考虑：
- 使用 `zhparser` 扩展
- 使用 Elasticsearch
- 使用专门的搜索服务

### Q2: 搜索性能如何优化？
**A**: 当前已有的优化：
- GIN 索引加速搜索
- 限制返回结果数量（10条）
- 缓存搜索结果（60秒）

进一步优化：
- 添加搜索结果缓存（Redis）
- 使用搜索建议/自动完成
- 分页加载结果

### Q3: 如何支持模糊搜索？
**A**: 当前使用 `plainto_tsquery`，支持：
- 多个词的 AND 搜索
- 自动处理空格

如需更灵活的搜索：
- 使用 `to_tsquery` 支持 OR、NOT 等操作符
- 使用 `LIKE` 或 `ILIKE` 进行模糊匹配
- 使用 `pg_trgm` 扩展进行相似度搜索
